# 리서치: 네이밍 데이터 저장 및 분류 체계 (v2.0)

## 1. 개요
브랜드 네이밍은 일회성 생성이 아닌 지속적인 고도화 과정이다. 결과물을 체계적으로 저장하고 분류하여 자산화할 수 있는 구조를 설계한다.

## 2. 저장 구조: 네이밍 세션 (Naming Session)

단일 브랜드명이 아닌, 하나의 기획 프로젝트를 '세션'으로 정의한다.

- **Session Meta:** 제목, 업종, 생성 일시, 작성자 ID.
- **Input Context:** 세션 생성 시 사용된 모든 분석 데이터 (경쟁사, 타겟, USP 등).
- **Outputs (Generated Names):**
    - 이름 (Name)
    - 논리적 근거 (Rationale)
    - 스타일/기법 (Style Tag)
    - 사용자 평가 (Score/Rating)
    - 보관 여부 (Saved/Archived)

## 3. 분류 및 탐색 체계 (Taxonomy)

### A. 시스템 자동 태깅
- **기법별:** #기술형 #추상형 #합성어 #두문자어
- **이미지별:** #모던한 #신뢰감있는 #창의적인 #심플한
- **언어별:** #순우리말 #영어 #한자어 #라틴어어원

### B. 사용자 커스텀 태깅
- 사용자가 프로젝트별로 자유롭게 태그를 추가하여 그룹화 가능.

## 4. 소셜 아키텍처 (Social & Collaborative)

### A. 개인 저장소 (Private Archive)
- 세션별 히스토리 무제한 저장.
- 특정 후보안에 대한 개인 메모 및 수정 이력 관리.

### B. 공유 게시판 (Public Gallery)
- **발행(Publish) 프로세스:** 세션 내에서 최종 후보로 낙점된 이름들만 선택하여 게시판에 공유.
- **공유 데이터 범위:** 이름, 논리적 근거, 사용된 핵심 키워드 (입력 데이터 중 민감 정보는 제외 가능).
- **상호작용:** 
    - **Inspiration Feed:** 타인의 로직과 이름을 보고 배울 수 있는 피드.
    - **Reaction:** 좋아요, 북마크(개인 보관함으로 가져오기), 댓글 피드백.

## 5. 결론: 기술적 구현 고려사항
- **DB:** 관계형 DB(PostgreSQL 등)를 통한 세션-이름-태그 간의 관계 관리 필요.
- **Auth:** 사용자별 데이터 격리 및 소셜 기능 구현을 위한 로그인 필수.
- **State Management:** Zustand를 넘어선 서버 상태 동기화(React Query 등) 고려.
